## 0. まず読む：このプロジェクトは何か

飲食店グループ（12店舗）の経営ダッシュボード。**最終ゴールは [flaro.jp](https://flaro.jp/function) のような統合ダッシュボード**（売上・広告・PL・人事・書類保管まで）。

- 本体: `index.html` + `app.js`（フレームワーク無しの素のJS。`app.js`は約6000行の1ファイル）
- バックエンド: `gas/Code.gs`（Google Apps Script。スプレッドシートがDB）
- 公開: https://mirai-oss.github.io/tori-dashboard/ （GitHub Pages。mainへpushで自動デプロイ・1〜2分）
- データ元スプレッドシート: ID `1iIQX6LqbM6rUrygfQZVGRVYtwEhO0O-5a_dvn06pWdA`

## 1. 触る前に知っておくべき「地雷」

### 🚨 GASの再デプロイはユーザーの手作業が必要
`gas/Code.gs` を変えたら、ユーザーがスプレッドシート側で貼り替えて再デプロイしないと反映されません。**手間をかけるので、GAS変更は必要な分をまとめて1回で済ませること。**

- 正しい手順: デプロイを管理 → **編集（鉛筆）→ バージョン「新バージョン」→ デプロイ**
- ❌「新しいデプロイ」を作るとURLが変わり、`app.js`の`DEFAULT_API_URL`が旧版を指したまま事故る
- 反映確認: `curl -sL '<GAS_URL>?action=ping'` → `ver` を見る（コード内の`ping`応答と一致すればOK）
- **GASを変えたら`ping`の`ver`も必ず上げること**（確認手段がなくなる）

### 🚨 `.github/workflows/` にはpushできない
GitHubトークンに`workflow`スコープが無く、pushが拒否されます。
→ **`scripts/lark-report.workflow.yml` を更新し、ユーザーにWeb UIで `.github/workflows/lark-report.yml` へ貼り替えてもらう**運用です。

### 🚨 push前に必ず `git fetch`
ユーザーは**別のClaudeセッションを並行で走らせることがあります**。リモートが先行していたら**rebaseで統合**（上書き厳禁。過去に複数回発生）。

### 🚨 ブラウザ検証時のキャッシュ
`app.js`を編集したら **`index.html`の`?v=N`を必ず+1**。上げないとブラウザが旧版を読み続け、「直したのに直らない」で時間を溶かします（実際に発生）。

### 🚨 ローカル検証の手順
```
python3 -m http.server 8766     # launch.jsonのport 8765は別セッションが専有しがち
```
ログインは本番APIを叩くので、検証時はブラウザのコンソールで：
```js
apiUrl=()=>'';                                    // API接続を切ってデモモードに
S.auth={token:null,account:{id:'shacho',name:'社長',role:'社長',stores:'全店',tabs:'',perms:'',position:''}};
render();
```
`sample-data.js` のデータは **2026-07-02頃まで**。それ以降の日付を見ると全部0になります（バグではない）。

### 🚨 スクリーンショットがスクロールを反映しない
Browser toolの`screenshot`は`window.scrollTo`を反映しないことがあります。パネルを一時的に`root`直下へ移動させてから撮ると確実です。

## 2. 現在の全体像

### タブ構成（`TAB_LABELS` / `ROLE_TABS` in app.js）
ダッシュボード / 目標管理 / 推移分析 / 明細分析 / PL（損益） / 入金管理 / 広告管理 / 口コミ / **週報** / **週報管理** / AI検索 / アカウント管理

### 権限モデル（3層構造。ここが分かりにくいので注意）

| 概念 | 値の例 | 何を決めるか | どこで設定 |
|---|---|---|---|
| **権限** (`role`) | 社長 / 本部 / マネージャー / 店舗 | データの見える範囲 | アカウントシートD列 |
| **役職** (`position`) | 店長 / 社員 | 週報フォーマット・権限の上書き | アカウントシートJ列 |
| **担当店舗** (`stores`) | 全店 / 店舗名カンマ区切り | 実際の店舗スコープ | アカウントシートE列 |

**「表示するタブ」「使える機能」の優先順位**（`myTabs()` / `myFeatures()`）:
```
アカウント個別設定 → DB_権限定義の役職行 → DB_権限定義の権限行 → コード内の既定(ROLE_TABS/ROLE_FEATURES)
```
- 「使える機能」= データを**書き込む**操作（口座CSV取込・経費入力・広告費入力・売上入力・予約CSV取込）
- 空欄＝上位の既定に従う / `なし`＝明示的に全部不可（**この区別が重要**。既定が「全部」の社長で全部外したとき既定に戻らないようにするため）
- ボタンを隠すだけでなく `App.open*` の入口でも `requireFeature()` で弾いている（開発者ツール対策）

### データの取り込み方（重要な設計）
- **シート名が「DB_」で始まるものはGAS再デプロイ不要で自動配信**される（`app.js`側の判定を足すだけで新シートを扱える）
- 新シートの**雛形自動作成**だけはGAS変更が必要
- 判定関数は `is○○Key()` で定義。**前方一致に注意**（例: `週報回答` は `週報` にも一致するので厳密一致 `/^週報$/` にしてある）

### 主なシート
| シート | 用途 |
|---|---|
| 分析_日別店舗 | 売上・客数・原価・人件費の日次（数値の正） |
| 分析_媒体別日次 | 媒体別売上（ランチ/ディナー内訳の元） |
| アカウント | ログイン情報（**10列**: ID/PW/表示名/権限/担当店舗/有効/メモ/表示タブ/使える機能/役職） |
| DB_権限定義 | 役職・権限ごとの既定（表示タブ・使える機能） |
| DB_週報テンプレート | 週報フォーマット（役職ごと。**行を足すだけで入力欄が変わる**） |
| DB_週報 / DB_週報回答 / DB_週報FB | 週報の提出・回答（縦持ち）・フィードバック |
| DB_招待 | 招待リンクのトークン |
| DB_PL / DB_広告 / DB_目標 / DB_イベント / DB_祝日 ほか | 各機能のデータ |

## 4. Lark日報の配信構成

1回の起動（GASの時計トリガーが正午にworkflow_dispatch）で以下が全部送られます。

| 宛先Secret | 内容 |
|---|---|
| `LARK_WEBHOOK` | 全店比較表 |
| `LARK_WEBHOOK2` | 鳥一代8店舗の比較表 |
| `LARK_WEBHOOK_GROUP1` | 本店 / はなれ / 芝 / 新橋 を**1店舗ずつ4枚** |
| `LARK_WEBHOOK_GROUP2` | 恵比寿 / 黒霧屋新横浜 / 鶏武者川崎店 を3枚 |
| `LARK_WEBHOOK_GROUP3` | 黒霧屋新横浜 / 鶏武者川崎店 / 鶏武者新横浜 を3枚 |

- **`App.report(kind, date, 店舗, group)` に店舗を1つだけ渡すと単店舗フォーマットに自動切替**（比較表→直近推移+ランチ/ディナー内訳+媒体別売上）
- Webhookのカスタムキーワードは「日報」。カード末尾のnote行に必ず入るので週報・月報でも通る
- **Webhook URLは絶対にリポジトリに書かない**（Publicリポジトリ）。GitHub Secretsのみ
- 手元から手動送信する手順は「作業ログ」2026-07-20の項を参照

## 5. 作業ログ

### 2026-07-20（大きく進んだ日）

**天気表示**
- Open-Meteo（APIキー不要）で実績＋16日先予報を取得。台風級の暴風・雷雨・大雨・大雪は色付きバッジ
- **この天気は実績ベース**。「台風予報だったが実際は来なかった日」に台風マークが付かないのは正常
- 日別明細・推移分析の天気をアイコンのみ→「アイコン＋天気名＋気温」に統一（日別予実と同じ形式）

**店舗別日報（Lark）**
- 単店舗フォーマットを新設。KPIに累計売上/累計F率/累計L率・Google口コミ増加・媒体別売上を追加
- ワークフローを`group-report`（1グループに複数店舗をループ送信）に組み替え。matrixに`店舗名|画像キー`をカンマ区切りで並べる
- 🐛 **修正**: `capture`が媒体データ到着前に撮影していた。媒体は裏読みで**約90秒後**に届くため、ランチ/ディナー内訳と媒体別売上が空で送信されていた（`D.mediaPending===false`を待機条件に追加）
- 実データで全8店舗ぶんの日報を手動送信して動作確認済み

**権限まわり**
- 入力・取込ボタン5種を「使える機能」で制御。社長がアカウント画面から設定可能に
- `DB_権限定義`シートを新設し、役職・権限ごとの既定をシートで管理できるように
- 🔐 **パスワードをSHA-256+個別saltでハッシュ化**。それまで平文でシートに保存されていた。従業員の自己登録を始めると数十人分の個人パスワードが平文で並ぶため、対象が4アカウントの今のうちに実施。**既存の平文はログイン成功時に自動でハッシュへ移行**するので再設定不要

**週報機能**
- 週の区切りは**火曜〜翌月曜**（分析タブの月内ブロック週とは別物・意図的）。提出=翌火16時 / FB=翌水16時
- 売上・客数・FL率・ダイニー点数はその人の店舗・その週で**自動集計**して入力欄の上に表示
- 閲覧は「**同じ店舗なら全員**」。上下関係の紐付けは作らず運用に委ねる方針（ユーザー合意済み）
- フィードバックは閲覧できる人なら誰でも書け、1つの週報に複数人が書ける
- 招待リンクによる自己登録（32桁トークン・1回限り・期限付き・一括発行1〜20枚）
- 社長向けに`週報管理`タブ（提出率・未提出者一覧・検索・フィルター）
- 🐛 **修正**: `wkTemplateFor`が役職未設定時に「社員」へ自動フォールバックし、社長にも投稿欄が出ていた

**ローカルから本番日報を送る手順（実績あり）**
```bash
npm install puppeteer --no-save
REPORT_KIND=daily DASH_ID=report DASH_PW=<PW> \
  REPORT_STORES='鳥一代 本店' REPORT_GROUP=honten node scripts/lark-report.mjs capture
gh release upload reports <fileKey>.png --clobber
LARK_WEBHOOK=<hook> IMAGE_URL=<releaseURL> node scripts/lark-report.mjs send
```
1店舗あたり約1〜2分（媒体データ待ちのため）。終わったら`node_modules`と`report.png`を消すこと。
